## 1. KPuzzle / KPattern Data Model

### What is KPuzzle?

A **KPuzzle** is an abstract representation of a twisty puzzle's mechanics. It defines:
- The puzzle's name
- Its orbits (groups of interchangeable pieces)
- The default solved state
- How each move transforms the puzzle state

For the 3Ă—3Ă—3 cube, the KPuzzle is created with:
```js
import { cube3x3x3 } from 'cubing/puzzles';
const kpuzzle = await cube3x3x3.kpuzzle();
```

The puzzle definition lives at:
`src/cubing/puzzles/implementations/dynamic/3x3x3/3x3x3.kpuzzle.json.ts`

### What is KPattern?

A **KPattern** is a mutable state snapshot of a puzzle. It contains:
- Reference to the KPuzzle definition
- `patternData`: an object mapping orbit names to orbit data

```typescript
interface KPattern {
  kpuzzle: KPuzzle;
  patternData: KPatternData;
}
```

### patternData Structure

`patternData` is a dictionary with one entry per orbit. For 3Ă—3Ă—3, there are three orbits:

```typescript
type KPatternData = {
  EDGES:   KPatternOrbitData;
  CORNERS: KPatternOrbitData;
  CENTERS: KPatternOrbitData;
};

interface KPatternOrbitData {
  pieces:          number[]; // Which home piece is at each position
  orientation:     number[]; // Orientation of the piece at each position
  orientationMod?: number[]; // (Optional) Orientation visibility mod
}
```

## 3. Piece and Orientation Values

### `pieces[i]`
- `pieces[i]` = the home slot ID of the piece currently occupying position `i`
- In solved state: `pieces = [0, 1, 2, ..., numPieces-1]`

Example: `EDGES.pieces[0] = 5` â†’ the piece that belongs at home position 5 (DR edge) is currently at position 0 (UF edge slot).

### `orientation[i]`
- **Corners**: 0, 1, or 2 â€” clockwise rotation count
- **Edges**: 0 or 1 â€” 0 = unflipped, 1 = flipped
- **Centers**: 0â€“3 â€” 90Â° rotation steps; normally invisible (see `orientationMod`)

### `orientationMod` (CENTERS only, normal 3Ă—3Ă—3)
```js
CENTERS.orientationMod = [1, 1, 1, 1, 1, 1]
```
A value of `1` means "known mod 1" â€” orientations are invisible (treated as 0 when checking solved state). On a supercube, omit this field to make center orientations visible.

## 5. Move Application

### API

```typescript
kPattern.applyMove(move: Move | string): KPattern
kPattern.applyAlg(alg: Alg | string): KPattern
kPattern.applyTransformation(transformation: KTransformation): KPattern
```

All three are immutable â€” they return a new KPattern.

### Internal Algorithm (from `combine.ts`)

For each orbit, for each position `i`:
```
newPieces[i]      = patternData.pieces[transformation.permutation[i]]
newOrientation[i] = (patternData.orientation[transformation.permutation[i]]
                     + transformation.orientationDelta[i]) % numOrientations
```

Key: **permutation first** (select which piece moves here), **then add orientation delta**.

### The Alg Class

```typescript
import { Alg } from 'cubing/alg';

new Alg("R U R' U'")    // parse from string
alg.invert()            // reverse order, each move inverted
alg.toString()          // serialize back to string
alg.expand()            // expand groupings/commutators to explicit moves
alg.isIdentical(other)  // structural equality check
```

**Inversion example:**
```js
new Alg("R U2 F'").invert().toString()  // â†’ "F U2' R'"
```

Inversion is purely syntactic â€” reverses move order and negates each move's amount.

## 6. Whole-Cube Rotations and CENTERS

Rotations (x, y, z, x2, y2, z2) affect both **permutation** and **orientation** in the CENTERS orbit.

| Move | CENTERS permutation | orientationDelta      | Net effect on centers |
|------|---------------------|-----------------------|-----------------------|
| x    | [2,1,5,3,0,4]       | [0,3,0,1,2,2]         | Uâ†’F, Fâ†’D, Dâ†’B, Bâ†’U   |
| y    | [0,2,3,4,1,5]       | [1,0,0,0,0,3]         | Fâ†’R, Râ†’B, Bâ†’L, Lâ†’F   |
| z    | [1,5,2,0,4,3]       | [1,1,1,1,3,1]         | Uâ†’R, Râ†’D, Dâ†’L, Lâ†’U   |
| z2   | [5,4,2,0,1,3]       | [2,2,2,2,2,2]         | Uâ†”D, Râ†”L, F and B unchanged |

**After z2 on a solved cube:**
```js
// CENTERS.pieces  = [5, 4, 2, 0, 1, 3]  (U slot has D piece, R slot has B piece, etc.)
// CENTERS.orientation = [2, 2, 2, 2, 2, 2]  (each center rotated 180Â°)
```

For sticker-colour derivation: the piece at U slot (index 0) is piece 5 (yellow/D). That's why the U face shows yellow after z2.

## 7. Sticker Colour Derivation

### Formula

Given a KPattern, to find the colour visible at face-slot `s` on the piece at position `i` in orbit `o`:

**Step 1** â€” identify the home piece:
```js
const homePiece = pattern.patternData[o].pieces[i];
const orientation = pattern.patternData[o].orientation[i];
const N = kpuzzle.definition.orbits[o].numOrientations;
```

**Step 2** â€” compute visible sticker index:
```js
const visibleSticker = (s - orientation + N) % N;
```

> âš ď¸Ź The formula is `(s - orientation + N) % N` â€” **NOT** `(s + orientation) % N`.

**Step 3** â€” look up the colour from the home piece's sticker table at index `visibleSticker`.

### Worked Example: URF corner after R move

Suppose after applying `R`, the URF corner slot (index 0) contains home piece 4 (DRF), orientation 2.

Home piece 4 (DRF) sticker colours in home-slot order: `[D, F, R]` = `[yellow, green, red]`

For sticker slot 0 (U face): `(0 - 2 + 3) % 3 = 1` â†’ colour = green (F)
For sticker slot 1 (R face): `(1 - 2 + 3) % 3 = 2` â†’ colour = red (R)
For sticker slot 2 (F face): `(2 - 2 + 3) % 3 = 0` â†’ colour = yellow (D)

## 8. `experimentalIsSolved()`

```typescript
pattern.experimentalIsSolved({
  ignorePuzzleOrientation: boolean,
  ignoreCenterOrientation: boolean
})
```

- **`ignorePuzzleOrientation: true`** â€” normalises the cube to its canonical U/R orientation before checking. Use this when whole-cube rotations should not count as "unsolved" (e.g. a z2-flipped solved cube is still "solved").
- **`ignoreCenterOrientation: true`** â€” zeroes all centre orientations before checking. Use on normal 3Ă—3Ă—3 since `orientationMod = 1` makes centre rotations invisible anyway.

Implementation converts the pattern to a KTransformation. If it equals the identity transformation (after optional normalisation), the puzzle is solved.

## 9. Key Gotchas

| Gotcha | Detail |
|--------|--------|
| **CENTERS slot order** | U=0, R=1, F=2, **L=3**, **B=4**, **D=5** â€” NOT URFDLB |
| **Orientation formula** | `(s - o + N) % N` not `(s + o) % N` |
| **z2 orientation tracking** | All 6 centres get `orientationDelta = 2`; on normal cube this is invisible but pieces[] still swaps |
| **Whole-cube rotations move CENTERS** | x, y, z affect both pieces[] and orientation[] in CENTERS orbit |
| **orientationMod = 1** | On normal 3Ă—3Ă—3, centre orientations are tracked internally but invisible; don't confuse with orientations being zero |
| **applyMove is immutable** | Always returns a new KPattern; the original is unchanged |
| **Alg.invert() is syntactic** | Just reverses and negates moves; puzzle-agnostic |
| **`experimentalIsSolved` name** | Marked experimental but stable in practice; `ignorePuzzleOrientation: true` is the flag to use for CFOP contexts where z2 setup is applied |
