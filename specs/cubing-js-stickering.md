## 1. Architecture: Why TwistyPlayer Can Show Yellow on Side Faces

### The fundamental difference from a slot-based renderer

TwistyPlayer's Cube3D uses **facelet-identity-based** stickering. Each physical sticker on each cubie is a persistent 3D mesh with a material baked from its **home face colour** at construction time. That material never changes. `onPositionChange` only moves the cubie's 3D group (position + orientation rotation) ÔÇö it never reassigns materials.

`setStickeringMask` controls which sticker meshes are visible (`regular` / `dim` / `ignored` / `invisible`), but always assigns materials from the **face-colour axis** of that sticker's home face, not from whatever face it currently occupies.

**Consequence for IgnoreNonPrimary ('O'):**
- facelet[0] of each piece always carries the primary face colour (white for U-layer pieces, yellow for D-layer pieces)
- When the piece is twisted, the entire cubie group rotates in 3D ÔÇö that sticker mesh physically moves to a side face
- It still carries yellow (or white) paint ÔÇö the viewer sees yellow on the side face
- The secondary/tertiary sticker meshes are set to `ignored` (grey)

This is why TwistyPlayer shows yellow on F/R/B for twisted OLL corners: the yellow sticker mesh *travels* to the side face.

### A slot-based renderer (like our harness) behaves differently

In our harness, `setState` reads the KPattern Ôćĺ face array and assigns materials to fixed Three.js material slots by grid position. After z2 baked in the KPattern, yellow appears at U-layer grid positions correctly. But `applyStickering` with 'O' always greys all slots *except slot 2 (+Y)*, regardless of orientation. For a twisted corner:
- Slot 2 (+Y) = the colour currently at the U face slot Ôćĺ non-yellow (e.g. red or green)
- Slot 0 or 4 = the yellow sticker Ôćĺ greyed out

## 2. Facelet Ordering Per Piece

### facelet[0] is always the first letter in the CubieDef name

```typescript
new CubieDef("CORNERS", "UFR", ...)  // facelet[0]=U, [1]=F, [2]=R
new CubieDef("CORNERS", "URB", ...)  // facelet[0]=U, [1]=R, [2]=B
new CubieDef("CORNERS", "DRF", ...)  // facelet[0]=D, [1]=R, [2]=F
```

| Piece | facelet[0] | facelet[1] | facelet[2] |
|-------|-----------|-----------|-----------|
| U-layer corners (UFR, URB, UBL, ULF) | U sticker | varies | varies |
| D-layer corners (DRF, DFL, DLB, DBR) | D sticker | varies | varies |
| U-layer edges (UF, UR, UB, UL) | U sticker | varies | ÔÇö |
| D-layer edges (DF, DR, DB, DL) | D sticker | varies | ÔÇö |
| Equatorial edges (FR, FL, BR, BL) | F or B sticker | varies | ÔÇö |

**For all U/D layer pieces, facelet[0] is the U or D face sticker** ÔÇö the "orientation sticker" in OLL/PLL context.

| Type | Colour |
|------|--------|
| `regular` | Face colour (white for U, yellow for D, etc.) |

| | `O` (OLL) | `P` (PLL) |

## 5. The Rendering Pipeline

### Phase 1: Construction (one-time per cubie)

```typescript
// For each facelet i of a piece:
sticker.material = axesInfo[piece.stickerFaces[i]].stickerMaterial.regular;
faceletInfo.faceIdx = piece.stickerFaces[i];  // stored permanently
```

Material is frozen to the face colour. `faceIdx` is stored for future mask lookups.

### Phase 2: Stickering mask application

```typescript
// For each piece slot in the mask:
for (let faceletIdx = 0; ...; faceletIdx++) {
  faceletInfo.facelet.material =
    axesInfo[faceletInfo.faceIdx].stickerMaterial[maskType]; // maskType = "regular"/"ignored"/etc
}
```

The colour comes from the sticker's **home face axis**, not its current face. Only the variant (regular/dim/grey/invisible) changes.

### Phase 3: Position update (`onPositionChange`)

```typescript
// Move cubie j to slot i's home position, then apply orientation rotation
this.pieces[orbit][j].matrix.copy(pieceDefs[orbit][i].matrix);
this.pieces[orbit][j].matrix.multiply(orientationRotation[orbit][orientation[i]]);
```

**No colour update.** The sticker meshes (with their baked materials) travel with the cubie. Orientation rotation physically moves the primary sticker mesh to a different face. The viewer sees it there.

## 6. Implications for a KPattern-Based Slot Renderer

To replicate `IgnoreNonPrimary` behaviour (show primary sticker wherever it is, grey others) in a system where materials are assigned by position (not identity), you need **state-aware masking**:

For each U-layer corner at homePos (x, 1, z) with KPattern orientation `o`:

- The primary sticker is at sticker position `o` in that corner's face order
- Face order for each U-layer corner (from CORNER_POSITIONS in CubeState.js):
  - URF (1,1,1):  [U=slot2, R=slot0, F=slot4]  Ôćĺ o=0Ôćĺslot2, o=1Ôćĺslot0, o=2Ôćĺslot4
  - URB (1,1,-1): [U=slot2, B=slot5, R=slot0]  Ôćĺ o=0Ôćĺslot2, o=1Ôćĺslot5, o=2Ôćĺslot0
  - ULB (-1,1,-1):[U=slot2, L=slot1, B=slot5]  Ôćĺ o=0Ôćĺslot2, o=1Ôćĺslot1, o=2Ôćĺslot5
  - ULF (-1,1,1): [U=slot2, F=slot4, L=slot1]  Ôćĺ o=0Ôćĺslot2, o=1Ôćĺslot4, o=2Ôćĺslot1

- Show only the primary Three.js slot, grey all other outward slots

The mapping from KPattern corner slot Ôćĺ homePos is via `CORNER_ORBIT_TO_CUBELET` + `CUBELET_POSITIONS`.

The key question: which KPattern corner slot corresponds to each homePos?
- KPattern slot 0 (URF) Ôćĺ cubelet 25 Ôćĺ homePos (1,1,1)
- KPattern slot 1 (URB) Ôćĺ cubelet 23 Ôćĺ homePos (1,1,-1)
- KPattern slot 2 (ULB) Ôćĺ cubelet 6  Ôćĺ homePos (-1,1,-1)
- KPattern slot 3 (ULF) Ôćĺ cubelet 8  Ôćĺ homePos (-1,1,1)

Implementation: `CubeStickering.fromOrbitStringWithState(str, kPattern)` ÔÇö returns a vis map where 'O' entries use orientation-derived primary slots instead of the fixed slot 2.
